PoliteDB includes **query safety modes** to help you avoid accidental changes on important databases.

You can switch modes from the lock icon in the connection toolbar. The icon color also reflects the active mode:

- Unlocked icon: default mode.
- Red lock icon: lock mode.
- Green lock icon: safe mode.

## Safety modes

PoliteDB currently supports three modes. Each mode is designed for a different level of risk:

- **Default mode**
  - Shows a confirmation prompt before sending SQL queries from the SQL editor.
  - Keeps editing features available in table views.
  - Best for normal development where you still want a quick "are you sure?" step.
- **Safe mode**: requires Touch ID authentication before:
  - Sending SQL queries.
  - Saving table edits or schema changes from the table UI.
  - Best when you still need write access, but want strong operator verification.
- **Lock mode**
  - Blocks mutating SQL statements.
  - Disables editing actions in table-oriented workflows.
  - Best for read-only sessions, audits, and incident investigations.

## How safe mode behaves

When you use **Safe mode**:

- Every SQL run requires Touch ID confirmation.
- Every save action for pending table changes requires Touch ID confirmation.
- If Touch ID authentication fails or is canceled, PoliteDB cancels the action and shows an error.

This behavior gives you a hard verification checkpoint before high-impact actions. You can still move fast, but sensitive operations require explicit biometric approval.

## How lock mode differs from safe mode

Safe mode and lock mode are often used in similar environments, but they solve different needs:

- **Safe mode** allows writes, but requires Touch ID before execution.
- **Lock mode** prevents write operations entirely in protected paths.

Choose safe mode when a trusted operator should be able to make controlled changes. Choose lock mode when the entire session must remain read-only.

## Scope and persistence

Safety mode is stored **by profile**, not just by the current tab.

In practice, this means:

- If you switch one tab to Safe mode, other tabs from the same profile use the same mode.
- Your chosen mode is restored for that profile in future sessions.
- When you reconnect with the same profile later, PoliteDB keeps your last selected safety mode.

This helps teams maintain consistent safety posture for specific environments, such as production, staging, or shared QA databases.

## Recommended usage

Use each mode intentionally based on environment risk:

- **Default mode**
  - Local development.
  - Short-lived test databases.
  - Fast iteration when mistakes are easy to recover.
- **Safe mode**
  - Production-like environments where writes are allowed.
  - Team-shared databases with higher operational risk.
  - Sessions where schema changes or data fixes may be necessary, but should be tightly controlled.
- **Lock mode**
  - Read-only data exploration.
  - Review sessions and demos.
  - Compliance, audit, or incident triage workflows.

## Operational guidance

Safety modes are one layer of protection. For complete safety, combine them with:

- Least-privilege database roles.
- Reliable backup and restore procedures.
- Query review and change management practices.

Safety mode reduces accidental actions in the client. It does not replace database-level permissions, transaction strategy, or backup policy.
